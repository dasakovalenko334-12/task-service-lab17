# Дымовой набор проверок для релиза (Smoke Release Tests)

**Проект:** Task Service (Лабораторная работа 17)  
**Цель:** быстрая проверка базовой работоспособности сервиса задач перед релизом.  
**Статус:** готов к использованию.  
**Исполнитель:** dasakovalenko334-12

## Критерии прохождения (Pass/Fail Criteria)

Сборка считается **STABLE (стабильной)**, если успешно пройдены все 10 проверок (SMK-01 — SMK-10).  
Если хотя бы одна проверка завершается с ошибкой — сборка **REJECTED (отклонена)**.

---

## Чек-лист проверок

### SMK-01: Проверка запуска приложения
**Цель:** убедиться, что сервис стартует без ошибок.  
**Шаги:**
1. Перейти в корневую директорию проекта.
2. Запустить сервис командой: `uvicorn main:app --reload`.
3. Дождаться сообщения `Uvicorn running on http://127.0.0.1:8000`.
**Ожидаемый результат:** сервис запущен, в консоли нет ошибок импорта или синтаксических ошибок. Статус: **UP**.

### SMK-02: Проверка эндпоинта здоровья (Health Check)
**Цель:** подтвердить, что приложение отвечает на базовые HTTP-запросы.  
**Шаги:**
1. Выполнить запрос: `curl http://localhost:8000/health`.
**Ожидаемый результат:** ответ `200 OK` с JSON: `{"status":"healthy"}`.

### SMK-03: Проверка списка задач (GET /tasks)
**Цель:** убедиться, что API отдаёт список задач (даже пустой).  
**Шаги:**
1. Выполнить запрос: `curl http://localhost:8000/tasks`.
**Ожидаемый результат:** ответ `200 OK` с JSON-массивом: `[]`.

### SMK-04: Проверка создания задачи (POST /tasks)
**Цель:** подтвердить возможность добавления новой задачи.  
**Шаги:**
1. Выполнить запрос:
   ```bash
   curl -X POST http://localhost:8000/tasks \
     -H "Content-Type: application/json" \
     -d '{"title": "Smoke Test Task", "description": "Test description"}'
### SMK-05: Проверка изменения статуса задачи (PATCH /tasks/{id})
**Цель:** подтвердить возможность обновления состояния задачи.
**Шаги:**
1.  Получить ID созданной задачи (из предыдущего шага SMK-04). Допустим, ID = `1`.
2.  Выполнить запрос:
    ```bash
    curl -X PATCH http://localhost:8000/tasks/1 \
      -H "Content-Type: application/json" \
      -d '{"status": "done"}'
    ```
**Ожидаемый результат:** Ответ `200 OK` с данными задачи, где `"status": "done"`. При повторном запросе `GET /tasks` задача отображается со статусом `done`.

### SMK-06: Проверка сохранения данных на диск
**Цель:** убедиться, что изменения сохраняются в файл (например, `tasks.json`).
**Шаги:**
1.  Проверить наличие файла `tasks.json` в папке проекта.
2.  Открыть файл и найти в нём запись с задачей "Smoke Test Task".
**Ожидаемый результат:** Файл существует. В нём валидный JSON. Статус задачи в файле — `done`.

### SMK-07: Проверка восстановления после перезапуска (Data Persistence)
**Цель:** подтвердить, что данные не теряются при остановке и запуске сервиса.
**Шаги:**
1.  Остановить сервис (Ctrl+C в терминале).
2.  Запустить сервис заново: `uvicorn main:app --reload`.
3.  Дождаться сообщения `Uvicorn running on http://127.0.0.1:8000`.
4.  Выполнить запрос: `curl http://localhost:8000/tasks`.
**Ожидаемый результат:** Сервис запустился без ошибок. В списке задач присутствует "Smoke Test Task" со статусом `done`. Данные успешно загружены из файла.

### SMK-08: Проверка обработки ошибочного ввода (Bad Request)
**Цель:** проверить, что сервис не падает при некорректных данных.
**Шаги:**
1.  Попытаться создать задачу с пустым названием:
    ```bash
    curl -X POST http://localhost:8000/tasks \
      -H "Content-Type: application/json" \
      -d '{"title": "", "description": "Test"}'
    ```
**Ожидаемый результат:** Сервис **не** создаёт задачу. Возвращает ошибку `400 Bad Request` с понятным сообщением (например, `"error": "Title cannot be empty"`). Существующие задачи не повреждены.

## Пояснение выбора критических проверок (BLOCK)

Ниже приведено обоснование, почему именно эти проверки имеют статус `BLOCK` и запрещают выпуск релиза при провале:

1.  **SMK-01 (Запуск) и SMK-02 (Health Check):** Если сервис не запускается или не отвечает на базовые запросы, он полностью неработоспособен. Пользователь не сможет взаимодействовать с продуктом. Риск: полная недоступность сервиса.
2.  **SMK-07 (Восстановление после перезапуска):** Если после рестарта сервиса данные исчезают, продукт теряет свою основную ценность — хранение состояния. Пользователь создаст задачу, сервис перезагрузится (например, из-за автообновления), и задача исчезнет. Риск: потеря данных пользователей.
