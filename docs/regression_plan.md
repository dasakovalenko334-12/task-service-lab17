# План регрессионного тестирования для Task Service (ЛР17)

**Статус:** Draft  
**Версия:** 1.0  
**Ответственный:** dasakovalenko334-12  
**Покрывает Issue:** #1  

## Цель
Гарантировать, что изменения в коде не сломали существующий функционал сервиса задач (CRUD, валидация, фильтрация, контракты API).

## Приоритеты
- **P1 (Critical):** Критический путь, без которого продукт не работает (создание, просмотр, удаление).
- **P2 (High):** Важные функции, влияющие на удобство и безопасность (фильтрация, валидация, статусы).
- **P3 (Medium):** Вспомогательные функции и негативные сценарии.

## Сценарии регрессии

### Критический путь (CRUD)
| ID | Сценарий | Приоритет | Ожидаемый результат | Риск при провале |
| :--- | :--- | :--- | :--- | :--- |
| REG-01 | Создание новой задачи (POST /tasks) | P1 | Задача создаётся, возвращается 201, ID задачи в теле ответа | Потеря возможности добавлять задачи |
| REG-02 | Получение списка задач (GET /tasks) | P1 | Возвращается JSON-массив, статус 200, сортировка по дате создания | Пользователи не видят список задач |
| REG-03 | Получение одной задачи по ID (GET /tasks/{id}) | P1 | Возвращается JSON задачи, статус 200 | Невозможность просмотра деталей задачи |
| REG-04 | Обновление задачи (PUT /tasks/{id}) | P1 | Задача обновляется, возвращается 200, поля изменены | Невозможность редактировать задачи |
| REG-05 | Удаление задачи (DELETE /tasks/{id}) | P1 | Задача удаляется, статус 204, при повторном запросе — 404 | Невозможность очистки списка задач |

### Контракты данных и валидация
| ID | Сценарий | Приоритет | Ожидаемый результат | Риск при провале |
| :--- | :--- | :--- | :--- | :--- |
| REG-06 | Проверка валидации пустого названия задачи | P2 | Статус 400, сообщение об ошибке, задача не создана | Сохранение некорректных данных в БД |
| REG-07 | Проверка валидации слишком длинного названия (>100 символов) | P2 | Статус 400, задача не создана | Переполнение полей БД, ошибки отображения |
| REG-08 | Проверка формата JSON при создании (невалидный JSON) | P2 | Статус 400, понятное сообщение об ошибке парсинга | Падение сервиса при некорректном запросе |
| REG-09 | Проверка обязательных полей (отсутствие поля `title`) | P2 | Статус 400, задача не создана | Нарушение целостности данных |

### Фильтрация и статусы
| ID | Сценарий | Приоритет | Ожидаемый результат | Риск при провале |
| :--- | :--- | :--- | :--- | :--- |
| REG-10 | Фильтрация по статусу `completed` | P2 | Возвращается только список завершённых задач | Пользователи не могут фильтровать задачи |
| REG-11 | Фильтрация по статусу `pending` | P2 | Возвращается только список незавершённых задач | Потеря видимости активных задач |
| REG-12 | Фильтрация по несуществующему статусу | P3 | Возвращается пустой список или 400 (в зависимости от реализации), сервис не падает | Нестабильность API при неправильных запросах |

### Негативные сценарии и граничные значения
| ID | Сценарий | Приоритет | Ожидаемый результат | Риск при провале |
| :--- | :--- | :--- | :--- | :--- |
| REG-13 | Запрос задачи с отрицательным ID (GET /tasks/-1) | P3 | Статус 404 или 400, понятное сообщение | Уязвимости, раскрытие внутренней логики |
| REG-14 | Попытка обновления несуществующей задачи | P3 | Статус 404, данные не изменены | Ложные срабатывания в UI |
| REG-15 | Массовое создание задач (нагрузка) | P3 | Все задачи создаются, нет таймаутов, статус 201 для каждой | Падение сервиса при реальной нагрузке |

## Покрытие требований
- **CRUD:** Покрыто сценариями REG-01 – REG-05.
- **Контракты данных:** Покрыто сценариями REG-06 – REG-09.
- **Фильтрация:** Покрыто сценариями REG-10 – REG-12.
- **Негативные сценарии:** Покрыто сценариями REG-13 – REG-15.

## Стратегия выполнения
1.  Выполнить все сценарии P1. Если хотя бы один падает — стоп, релиз запрещён.
2.  Выполнить сценарии P2. Если есть ошибки — зафиксировать, но релиз возможен с оговорками.
3.  Выполнить сценарии P3. Ошибки фиксируются как бэклог.
